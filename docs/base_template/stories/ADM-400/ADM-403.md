# ADM-403: Site Settings Management

**Epic:** ADM-400 Admin Application  
**Status:** ✅ Done  
**Priority:** P1

---

## Story

> As an admin, I want a Settings page where I can view and update global site configuration stored in the Firestore `settings` collection so that I can manage theme preferences, feature flags, and environment-specific values without a code deploy.

---

## Acceptance Criteria

- [x] A `/dashboard/settings` route is added to `pages.routes.ts`.
- [x] `SettingsComponent` (`pages/settings/settings.component.ts`) loads settings from Firestore.
- [x] A `SettingsService` wraps Firestore reads/writes for the `settings` collection.
- [x] The settings form includes the following fields (initial scope):
  - **App Name** (text input)
  - **Theme** (select: `orange_theme` | `blue_theme` | `purple_theme`)
  - **Maintenance Mode** (toggle/checkbox)
  - **Feature Flags** section — at minimum one flag: `enable_registration` (boolean toggle)
- [x] Saving the form writes to Firestore `settings/global` document.
- [x] Unsaved changes show a **"You have unsaved changes"** banner.
- [x] A **"Save"** button is disabled until the form is dirty.
- [x] A **"Reset"** button reverts to last saved values.
- [x] Navigation item **"Settings"** is added to `sidebar-data.ts` under an `Admin` nav cap.
- [x] All fields have `data-test-id` attributes (e.g., `settings-app-name-input`, `settings-save-button`, `settings-maintenance-toggle`).

### Implementation Notes
- Fixed: `effect()` guard added (`isDirty` check) to prevent Firestore real-time emissions from overwriting unsaved user edits.
- Fixed: `setDoc()` wrapped in `runInInjectionContext()` to resolve AngularFire injection context warning.
- Fixed: `projectId` in `environment.local.ts` aligned to `novus-flexy` to match emulator namespace.

---

## Firestore Schema

Collection: `settings`, Document: `global`

```
settings/global
  ├── appName: string
  ├── theme: string
  ├── maintenanceMode: boolean
  └── featureFlags: {
        enable_registration: boolean
      }
```

---

## Technical Notes

- Use `docData(doc(firestore, 'settings', 'global'))` for real-time sync.
- Apply `firestore.rules` so only `admin` role can write to `settings`.
- The `SettingsService` should expose a Signal for current settings: `readonly settings = toSignal(settingsDoc$)`.
- Changes should apply immediately in the UI via Signal update, then persist to Firestore.
