## Story

## Context

Firebase Emulators run as a mixed set of processes:

| Emulator    | Runtime                                    | Port |
| :---------- | :----------------------------------------- | :--- |
| Auth        | Node.js (built into firebase-tools)        | 9099 |
| Firestore   | JVM â€” `cloud-firestore-emulator-vX.jar`    | 8085 |
| Functions   | Node.js                                    | 5001 |
| Storage     | JVM â€” `cloud-storage-rules-runtime-vX.jar` | 9199 |
| Emulator UI | Node.js                                    | 8080 |

Firestore and Storage emulators are Java `.jar` files. **Java 21+ must be installed**
in the container for these two emulators to start. Auth, Functions, and the UI run on
Node.js and do not require Java.

## Acceptance Criteria

- [ ] `devcontainer.json` installs Java 21 via the official devcontainers feature
- [ ] `devcontainer.json` sets `hostRequirements` to 4-core / 16GB (minimum for running
      both Angular dev servers + emulators concurrently without lag)
- [ ] Emulator `.jar` files are downloaded at container creation time (avoids 60s download
      on first `npm run dev`)
- [ ] `npm run firebase:emulators` starts all four emulators without errors
- [ ] Emulator UI is accessible at `http://localhost:8080`
- [ ] Both Angular apps connect to emulators (Auth + Firestore calls succeed via `environment.local.ts`)
- [ ] `npm run dev` starts all three processes together successfully

## Implementation

### devcontainer.json changes

**Java 21 feature added:**

```json
"ghcr.io/devcontainers/features/java:1": {
  "version": "21",
  "installMaven": false,
  "installGradle": false
}
```

**Host requirements set:**

```json
"hostRequirements": {
  "cpus": 4,
  "memory": "16gb",
  "storage": "32gb"
}
```

**Pre-download emulator JARs at container creation:**

```
npm install && npx firebase setup:emulators:firestore && npx firebase setup:emulators:storage
```

These JARs are cached at `~/.cache/firebase/emulators/` and are not re-downloaded on subsequent starts.

### Why `updateContentCommand` runs `npm install`

`onCreateCommand` runs once when the container is first created.  
`updateContentCommand` runs when the repo content changes (e.g. a `git pull` that changes `package.json`).  
The JAR downloads only need to happen once so they stay in `onCreateCommand`.

## Codespaces Machine Type

When creating or rebuilding this Codespace, select **4-core (16GB)** or higher.

The `hostRequirements` in `devcontainer.json` will prompt Codespaces to suggest the correct machine type.
You can also set it manually in the Codespaces UI before creation.

**Resource profile when running all three processes:**

| Process                            | Memory                        |
| :--------------------------------- | :---------------------------- |
| `ng serve admin`                   | ~300MB                        |
| `ng serve user-app`                | ~300MB                        |
| Firebase JVM (Firestore + Storage) | ~400â€“600MB                    |
| Firebase Auth + Functions + UI     | ~150MB                        |
| **Total**                          | **~1.2â€“1.5GB** with GC spikes |

A 2-core / 8GB machine is marginal. JVM startup competes with Angular's esbuild startup,
causing both to be slow. 4-core gives each process headroom without contention.
