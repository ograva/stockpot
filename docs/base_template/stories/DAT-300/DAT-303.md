# DAT-303: StoreForwardService — Serialize/Deserialize Transforms

**Epic:** DAT-300 Data Management
**Status:** ✅ Done
**Priority:** P1 — Depends on DAT-302 (Model Interfaces)

---

## Story

> As a developer, I want `StoreForwardService` to accept optional serialize and deserialize transform functions per call site, so that null sanitisation and schema migration logic lives with the model — not scattered across every component that writes to Firestore.

---

## Acceptance Criteria

- [x] A `StoreForwardTransforms<T>` interface is exported from `store-forward.service.ts`:
  ```typescript
  export interface StoreForwardTransforms<T> {
    serialize?:   (data: T)      => Record<string, unknown>;
    deserialize?: (raw: unknown) => T;
  }
  ```
- [x] `set<T>(key, value, firestorePath?, transforms?)` — updated signature:
  - If `transforms.serialize` is provided, it is called before the Firestore `setDoc()`. The serialized payload (not the raw value) is written to Firestore.
  - `localStorage` always stores the **full typed value** (pre-serialize) so cache reads are correctly typed.
- [x] `sync<T>(key, firestorePath, transforms?)` — updated signature:
  - If `transforms.deserialize` is provided, it is called on every incoming Firestore snapshot before the signal and `localStorage` are updated.
- [x] All existing call sites with no transforms continue to work identically — fully backward compatible.
- [x] `ProfileComponent` updated to pass transforms to both `set()` and `sync()`:
  ```typescript
  this.storeForward.sync<UserProfileDoc>(PROFILE_KEY, `users/${uid}`, {
    deserialize: deserializeUserProfile,
  });
  this.storeForward.set(PROFILE_KEY, profileDoc, `users/${uid}`, {
    serialize:   serializeUserProfile,
    deserialize: deserializeUserProfile,
  });
  ```
- [x] `ProfileComponent` no longer contains any manual null-stripping logic — all sanitisation delegated to `serializeUserProfile()`.
- [x] `ProfileComponent` is fully typed against `UserProfileDoc` — no more `Record<string, string | null>` intermediaries.

## Technical Notes

### Transform contract

```
                   ┌─────────────────────────────────────┐
  set(key, value)  │  1. signal.set(value)                │  ← full typed value
                   │  2. localStorage.setItem(serialise?) │  ← full typed value (not serialized)
                   │  3. Firestore setDoc(serialize(value))│  ← null-free Firestore payload
                   └─────────────────────────────────────┘

  sync(key, path)  │  onSnapshot(raw)                     │
                   │    → deserialize(raw) → signal.set() │  ← normalised, defaults filled
                   │    → localStorage.setItem(deserialized)│
```

### Why transforms belong on the call site, not baked into the service

`StoreForwardService` is a generic mechanism; it has no knowledge of document shapes. Attaching the model's own serialize/deserialize functions at the call site keeps the service reusable for any document type added in the future without modifying the service itself.

### Schema versioning via `_schemaVersion`

- `_schemaVersion` is written by `serialize` on **every** Firestore save.
- `deserialize` reads `_schemaVersion` and runs migration branches for any document older than the current version.
- Old documents are migrated **lazily** on the next read — no batch Cloud Function migration required at template scale.
