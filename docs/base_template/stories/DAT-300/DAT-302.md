# DAT-302: Firestore Document Model Interfaces

**Epic:** DAT-300 Data Management
**Status:** ✅ Done
**Priority:** P1 — Blocks DAT-303 (StoreForward Transforms)

---

## Story

> As a developer, I want typed interface files for every Firestore document I write, so that TypeScript enforces the correct shape at every call site and I can never accidentally introduce mismatched fields between the application and the database.

---

## Acceptance Criteria

- [x] A `models/` directory exists at `src/app/models/` within each app that writes to Firestore.
- [x] Every Firestore collection has exactly one canonical model file following the naming convention `{collection-name}.model.ts`.
- [x] Each model file exports:
  - A `SCHEMA_VERSION` constant (`number`) — e.g., `USER_PROFILE_SCHEMA_VERSION`.
  - A **Firestore document interface** (name ends in `Doc`) containing all stored fields plus `_schemaVersion: number` as the first field.
  - A **UI-facing type** when the Firestore document type should not be exposed directly (e.g., `SiteSettings = Omit<SiteSettingsDoc, '_schemaVersion'>`).
  - A `deserializeXxx(raw: unknown): XxxDoc` function — normalises raw Firestore snapshots, fills default values, and runs schema migrations.
  - A `serializeXxx(doc: XxxDoc): Record<string, unknown>` function — produces a `null`-free Firestore-safe payload.
- [x] `UserProfileDoc` model implemented at `projects/user-app/src/app/models/user-profile.model.ts`:
  - Fields: `_schemaVersion`, `name`, `email`, `role?`, `photoURL?`
  - `photoURL` is **never written** to Firestore when absent (omitted, not `null`).
- [x] `SiteSettingsDoc` model implemented at `projects/admin/src/app/models/site-settings.model.ts`:
  - Fields: `_schemaVersion`, `appName`, `theme`, `maintenanceMode`, `featureFlags`
  - `SiteSettings` UI type exported (strips `_schemaVersion`).
  - `SITE_SETTINGS_DEFAULTS` constant exported (used as signal `initialValue`).
- [x] `SettingsService` updated to import from the model file; inline `SiteSettings` interface and `defaults` constant removed from the service.
- [x] `SettingsComponent` import path unchanged (`SiteSettings` re-exported from `SettingsService` for backward compatibility).
- [x] No `null` values are ever passed to `setDoc()` or `updateDoc()` in any service.

## Technical Notes

### Model file contract (three-export pattern)

```typescript
export const MY_DOC_SCHEMA_VERSION = 1;

export interface MyDoc {
  _schemaVersion: number;   // always first; never stripped by serialize
  requiredField: string;
  optionalField?: string;   // never null — omitted when absent
}

export function deserializeMyDoc(raw: unknown): MyDoc {
  const data = (raw ?? {}) as Partial<MyDoc>;
  const version = data._schemaVersion ?? 0;
  if (version < MY_DOC_SCHEMA_VERSION) { /* migration branches */ }
  return {
    _schemaVersion: MY_DOC_SCHEMA_VERSION,
    requiredField: data.requiredField ?? 'default',
    ...(data.optionalField ? { optionalField: data.optionalField } : {}),
  };
}

export function serializeMyDoc(doc: MyDoc): Record<string, unknown> {
  const out: Record<string, unknown> = {
    _schemaVersion: doc._schemaVersion,
    requiredField: doc.requiredField,
  };
  if (doc.optionalField) out['optionalField'] = doc.optionalField;
  return out;
}
```

### Why interfaces over classes

- **Zero runtime cost** — interfaces are erased at compile time; no bundle impact.
- **Extendable** — `interface V2 extends V1` cleanly adds fields for schema bumps.
- **Structural compliance** — TypeScript enforces shape across every service, component, and test that touches the document.
