# TST-905: E2E Auth Infrastructure

## Story Description

As a QA engineer, I want a shared authentication infrastructure for the E2E suite — including an emulator seed script, a reusable login helper, and Playwright `globalSetup` wiring — so that any test requiring an authenticated session can rely on a single, consistent mechanism rather than duplicating login logic.

## Acceptance Criteria

- [ ] `e2e/global-setup.ts` seeds the Firebase Auth emulator with `admin@test.com / password123` before the suite runs.
- [ ] `global-setup.ts` handles `EMAIL_EXISTS` gracefully on re-runs (no failure, logs "already exists").
- [ ] `global-setup.ts` handles `ECONNREFUSED` gracefully when the emulator is not running (warns but does not throw, so individual tests fail with clearer messages).
- [ ] `global-setup.ts` skips seeding entirely when `TEST_ENV !== 'local'`.
- [ ] `TEST_USER` credentials are exported from `global-setup.ts` as the single source of truth — no hardcoded strings in spec files.
- [ ] `e2e/helpers/auth.helper.ts` exports `loginAsTestUser(page)` which drives the UI login flow and waits for the dashboard redirect.
- [ ] `playwright.config.admin.ts` declares `globalSetup: './e2e/global-setup.ts'`.
- [ ] No API keys or secrets are hardcoded — emulator accepts any non-empty string as the key.

## Definition of Done

- [ ] `e2e/global-setup.ts` exists and is wired into `playwright.config.admin.ts`.
- [ ] `e2e/helpers/auth.helper.ts` exists and is imported by T002 and T010 specs.
- [ ] Running `npm run test:e2e:admin` against live emulators produces a seeded test user before any spec file executes.
